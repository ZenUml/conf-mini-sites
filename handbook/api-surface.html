DeliveryAPI Surface

Every backend HTTP surface in one table — serving, publish, lifecycle, reconcile, OAuth/PAT and health — each with the auth it demands and the invariant it must honor. The dispatch Worker is the single network entry point and the auth gateway; there is no second door.

⚑
Status: BUILD GATED. This is design documentation, not shipped code. The classic-Connect verification path is net-new, greenfield, security-critical code — the single biggest reason a Cloud Fortified review is required, not skipped. Routing is deny-by-default: a new, unlisted route serves nothing until it is explicitly bound to a verified-auth handler (INV-GW-09).
⛨
The gateway is the CVSS-9.1 component. On the Cloudflare path the per-request permission/check is the only thing between an authenticated user and any bundle's bytes — one skipped check on any asset path is a cross-tenant content-disclosure breach (CVE-2021-26073 class), not a degraded feature. Every protected byte, including sub-resources, must be provably authorized.

⇥
Publish is async: 202 then poll, not a blocking 200. Spinning up a user Worker — WfP script upload plus a Static-Assets upload of many or large files plus a reachability smoke-check — can take minutes and can exceed a single Worker request's wall-clock budget. So POST /api/instances and PUT …/bundle validate + stage, enqueue a ProvisionInstance durable workflow, and return 202 {jobId, statusUrl}. The client (CLI or upload widget) then polls GET /api/jobs/<jobId> until state reaches active or failed. The instance stays staging and un-servable until the workflow's final atomic step flips it to active (I10) — the gateway never serves a staging/provisioning instance. The job manager lives above the HostingProvider seam, so it survives a Cloudflare→Forge pivot unchanged (§3.4).
⇥
Token extraction is canonical and strict. Entrypoint iframe GET → token from ?jwt= only. XHR/asset requests → Authorization: JWT <token> where the scheme must equal JWT (reject Bearer/other). If both ?jwt= and Authorization are present with different values → reject (parameter pollution). Do not inherit conf-app's scheme-blind split(' ')[1].

Deny-by-default routing

This inverts conf-app's _middleware.ts, which runs auth only for an AUTHENTICATED_PATHS allowlist and skips everything else — a fail-open posture. Here the allowlist holds exemptions only (/health, the signature-verified lifecycle and cron endpoints); everything else is auth-required by construction, and any unmatched or net-new route returns 401 before it can ever touch a bundle.

Typed error codes

Each upload-pipeline failure maps to one distinct, actionable error surface (and one analytics label, so we can measure where authors bounce). Auth/authz failures are deliberately opaque — they do not leak whether a resource exists, to avoid an existence oracle.

CodeStageMeaningDisposition
BUNDLE_NOT_MULTIFILE 3 — validation A single-file .html was submitted. Out of scope ("use the existing HTML macro"). Hard fail; rejected client-side for UX, re-checked server-side.
MISSING_INDEX_HTML 3 — validation The bundle has no root index.html entrypoint. Hard fail; nothing persisted.
ABSOLUTE_PATH_REJECTED 3 — validation A manifest path is absolute (http(s):// or a leading /) — not relative. Hard fail; same constraint shape as ATTACHMENT_NAME_RE, extended to a full manifest.
PATH_TRAVERSAL_REJECTED 3 — validation A manifest path contains .. (directory-traversal attempt). Hard fail; nothing persisted.
TOO_MANY_FILES 4 — size/count caps The bundle exceeds the server-authoritative max file count. Hard fail.
BUNDLE_TOO_LARGE 4 — size/count caps Per-bundle or per-file size cap exceeded (incl. decompressed total / zip-bomb ratio guard). Hard fail.
SECRET_DETECTED GATED ruleset 5 — secret-leak scan A high-signal secret (cloud key, bearer token, PEM header, .env assignment) or high-entropy hit was found. Hard fail; no bytes persisted (I7).
COMMIT_FAILED_ROLLED_BACK 7 — atomic commit The atomic stage→live commit failed; the partial upload was rolled back. A staging row remains that is never served and is GC'd (I10).
PROVISION_FAILED async provisioning (§3.4) The ProvisionInstance workflow exhausted its per-step retries (script upload, asset upload, or the dispatch smoke-verify never came back reachable). Surfaced as the error field on GET /api/jobs/<jobId> when state = failed. No servable bytes; the instance stays staging and is reaped by the orphan reaper after a TTL (I10 ↔ I1).
401 Unauthorized opaque Authn (§2) Token missing / malformed / bad alg / bad signature / expired / qsh mismatch / unknown iss. Carries a re-auth signal. Fail closed before the permission check; the 401-refresh loop is bounded only by the serving-request rate limit (INV-GW-15).
403 Forbidden opaque Authz (§2.5) permission/check returned false, or a registry/permission outage means ownership cannot be confirmed. Fail closed — deny even if a cached allow exists; the circuit-breaker default state is DENY. A cloudId mismatch returns 404, not 403, to avoid an existence oracle (I8).
⚑
Secrets and full tokens are NEVER logged — this is a structurally enforced invariant (lint rule + test), because the reused conf-app code violates it: authenticate.ts:28 logs the decoded token and forge-installed.ts:34 logs the full lifecycle body — and for classic Connect that body contains the sharedSecret (INV-GW-13). The per-decision audit log records {ts, clientKey, cloudId, accountId, instanceId, contentId, decision, reason, ruleId, jwtExp, latencyMs} and nothing more.

Publish pipeline ordering

The CLI, the MCP tool, and the drag-and-drop widget all call the same provider.createInstance() / updateBundle() behind the HostingProvider seam, so a Cloudflare→Forge pivot is a provider swap, not a publish-path rewrite. The pipeline is the security choke point: a bundle that fails any step never reaches status=live.

Steps 1–6 run inline in the publish request and decide whether to accept the bundle; step 7's staging is also inline. But provider.createInstance()/updateBundle() — the actual WfP script + asset upload and the dispatch smoke-check — can take minutes, so it does not run in the request. The request returns 202 {jobId, statusUrl} and a ProvisionInstance durable workflow performs upload script → upload static assets → smoke-verify reachable via dispatch → flip the instance to active, with per-step retries and backoff. A ProvisioningJob row tracks state (queued, provisioning, active, failed) and the client polls it at GET /api/jobs/<jobId>. The job is idempotent on Idempotency-Key + bundleHash; the orphan reaper reaps jobs stuck in provisioning past a TTL. Because the job manager sits above the HostingProvider seam, the same async orchestration drives ForgeProvider provisioning (custom-content + attachment chunking, also long-running) after a pivot.

Cost-bearing surfaces

The "~$25/mo, solo-buildable" premise prices only hosting. The acceptance criteria add a per-view and per-publish variable layer the premise ignored, and several of the endpoints above carry that load.

SurfaceDriverCost direction
GET /v/<instanceId>/g/<grant>/* D1 lookup + lastSeenAt + permission/check cache per protected view COST Grows with views; tens → low-hundreds $/mo of D1 + KV + Worker CPU.
Audit of every serve / deny / publish / delete R2 storage + Class-A writes (§5.6/I9) COST Unbounded if retention is loose; a dominant line at scale.
POST /api/instances · PUT …/bundle Secret-scan compute per uploaded bundle (CPU-heavy, spiky) COST Grows with publish volume; co-dominant with audit at 10k instances.
▤
Honest re-costing. Cheap to host, expensive to make safe and compliant. A realistic loaded run-rate at 10k active instances is order ~$500–700/mo (audit retention + secret-scan dominate), and the engineering cost — the greenfield auth gateway, a recurring Cloud Fortified review + external pen-test (a four-figure recurring item), orphan GC, provable DSAR erasure — dwarfs the infra. This belongs in the EV-vs-kill-criterion gate (#4).

What is GATED on this surface

  • OAuth client registration + scopes for the Flow A device grant (write:confluence-content, read:confluence-content.permission, custom-content write) — a vendor capability to confirm before building Flow A. Until confirmed, Flow A ships token-only (/oauth/login is gated).
  • Secret-scan ruleset (the SECRET_DETECTED source) is provisional — final patterns + entropy thresholds + allow-list are not pinned.
  • The exact signed claim that yields contentId for the §2.5 bind on the serving endpoints; if none exists for this iframe type, re-derive from the macro localId carried in qsh.
  • AVI event reliability for eager-fork (I2) and event-driven permission-change cache invalidation (I3) — both would tighten the serving and reconcile surfaces, both currently gated.
