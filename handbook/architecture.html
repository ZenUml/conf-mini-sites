DesignArchitecture & Components

How a user-uploaded multi-file static bundle is hosted on Cloudflare and served, live and isolated, inside a Confluence page — the Workers-for-Platforms dispatch model, the component topology, the tenant lifecycle, and the HostingProvider seam that keeps a forced Forge pivot bounded rather than catastrophic.

⚑
Design only — build gated. No production code ships until all four blocking gates pass (residency-vs-access sign-off; demand from ≥3 prospects beyond the n=1 anchor; signed lifecycle/security acceptance criteria; EV-vs-next-feature + kill criterion). This page is design documentation, not an as-built record.

Global constraints (every section inherits these)

Three settled facts shape every component below. They are not relitigated here — they are the boundary conditions the architecture must satisfy.

NOT no-egress Must pass Cloud Fortified WfP REQUIRED Honest re-cost ~$500–700/mo
▲
Files live on Cloudflare, not inside Atlassian. This is NOT "Runs on Atlassian", NOT no-egress, NOT in-boundary. Permissions are app-enforced ACL code, not inherited from Confluence. Consequence: the app must pass (not skip) a Cloud Fortified security review, and the auth gateway is a self-built ACL of the CVE-2021-26073 / CVSS-9.1 class — threat-modeled and pen-tested before any production traffic.
◆
Workers for Platforms is required, not optional. Plain Workers (500/account) and Pages (100/account) script caps make per-instance isolation impossible at scale. Per-instance isolation is the product's security primitive, so WfP is a hard dependency.
⚑
The Cloudflare→Forge pivot must be a provider swap, not a rewrite. If gate #1's residency review forbids an external processor, Cloudflare is disqualified; the design must survive that flip without a catastrophic rewrite. [hardened] §6 below corrects an over-optimistic reuse estimate — the pivot is bounded, not free.

The WfP model: one dispatch namespace, one user Worker per macro instance

The hosting substrate is a single dispatch namespace per environment (mini-sites-dev|stg|prod), mirroring conf-app's per-env wrangler-*.toml split verified. The dispatch Worker holds the namespace binding and is the single auth gateway; every macro instance gets its own non-routable user Worker inside that namespace, whose only job is to serve its own bundle via Workers Static Assets.

The user Worker serves index.html plus sibling app.js, style.css, assets/* exactly as on a normal static host. This native relative-path serving is the property that distinguishes a multi-file bundle from the single-file .html the existing HTML macros already handle — and it is the product's core capability.

Non-routability is the security primitive — and it is ENFORCED, not assumed hardened

⛨
User Workers in a dispatch namespace have no public route, no *.workers.dev subdomain, and no custom domain, and are unreachable by cross-tenant Worker-to-Worker fetch. They are reachable only via env.MINISITES.get(workerName).fetch() from the dispatch Worker. The whole ACL rests on this — so it is a config policy + a pen-test obligation, not a platform assumption. Deployment config asserts no route/subdomain/domain on dispatched scripts, and the pen-test must include an unauthenticated direct probe of a user Worker URL (INV-GW-14).

Naming / identity scheme

instanceId sha256(cloudId + ":" + macroLocalId) truncated to a Worker-name-safe slug. Stable primary key for one mini-site on one page. The Connect per-macro localId + cloudId is globally unique and survives reload — but not page copy (owned by the Lifecycle model, §5.1/I2).
workerName ms-<instanceId> — the script name inside the dispatch namespace. Never exposed to the client. The iframe URL only ever targets the dispatch Worker; the instanceId → workerName mapping is server-side only. Guessing a workerName buys an attacker nothing — no public route resolves it.

Request flow, end to end

There is exactly one network entry point: the dispatch Worker. Every byte — the initial index.html and every relative sub-resource — re-enters the gateway, which re-validates the grant and re-checks the permission cache before dispatching into the non-routable user Worker.

The premise does not survive intact. Cheap to host, expensive to make safe and compliant. A realistic loaded run-rate at 10k active instances under real traffic is order ~$500–700/mo (audit retention + secret-scan dominate), and the engineering cost dwarfs the infra: the greenfield auth gateway (CVSS-9.1 class), a recurring Cloud Fortified review + external pen-test (a four-figure recurring item, not a rounding error), orphan GC, and provable DSAR erasure are each multi-week security-critical workstreams. This belongs in the EV-vs-kill-criterion gate (#4).

The HostingProvider seam + Forge-pivot insulation

Two distinct host abstractions are kept separate so a forced pivot is bounded. The Confluence side (IConfluenceHost, reusing conf-app's IApWrapper shape verified) abstracts macro render, content reads, permission checks, and metadata storage. The hosting side (HostingProvider, new and load-bearing) abstracts where bundle bytes live, how they're served, and how a viewer is authorized — conf-app has no analog, since it stores only diagram DSL with no large-asset hosting substrate.

The seam contract

◆
No module above src/hosting/ imports a Cloudflare or Forge SDK. The interface exposes createInstance (idempotent on id), updateBundle (atomic from the viewer's POV), deleteInstance (idempotent — GC calls blind), serve(handle, filePath, auth), and verifyHostToken (WfP: Connect HS256/qsh; Forge: invocation JWKS). It also exposes a permissionModel flag ('app-enforced' | 'inherited') that drives which invariants apply, and a capabilities descriptor.

Above the seam (provider-agnostic, reused on pivot): macro registration & render; the upload pipeline (unzip, manifest, CSP rewrite, secret-scan, sandbox policy); the auth-gateway orchestration (verify + permission/check); lifecycle reconciliation (orphan/copy/export/DSAR/partial-failure); the invariant test suite (runs against the interface via a fake + a contract-test pack each provider must pass). Below the seam (rewritten on pivot): CloudflareWfPProvider (dispatch namespace, user Workers, Static Assets, D1 mapping, wrangler-*.toml); ForgeProvider (gated).

Forge fallback reality check — NOT free hardened

▲
serve() is NOT a no-op behind a flag. Under Forge it relocates the entire serving mechanism into a new client-side reassembly engine on the security-critical render path — fetch-all-files + relative-path rewriting (script src, CSS url()) via blob:/service-worker + CSP-on-blob-origin handling. That is net-new, security-critical code that the ≥80%-reuse figure must exclude — not a one-line supportsServerSideServe=false.

No server-side static servingserve() cannot stream files; the bundle must be reassembled client-side in the iframe (fetch all files + blob-URL/service-worker relative-path rewrite).

⚑
Bottom line. Forge does not make the security problem disappear — it relocates it (gateway CVE-risk down; client-reassembly + hard storage caps up). The feasibility spike must prove reassembly + relative-path rewriting + ~100MB/file chunking + CSP-on-blob viability on real bundles before a single ForgeProvider line is written.

The CI import boundary is NEW and must be machine-enforced hardened

⛨
The original design claimed the boundary is enforced "exactly as conf-app keeps @forge/bridge confined." That is false [verified by red team]: conf-app imports @forge/bridge from 15+ non-test modules, has no no-restricted-imports/boundary lint rule, and its eslint.config.mjs ignores functions/ (the entire backend). conf-app is live evidence the discipline rots without a gate.
InvariantWhat it enforces
INV-SEAM-01A CI-enforced architectural-boundary check (eslint no-restricted-imports or dependency-cruiser) fails the build on any @cloudflare/* or @forge/* import above src/hosting/, and must lint functions/ (do not inherit conf-app's ignore). Built from day one — the ≥80% reuse number is unprotected without it.
INV-SEAM-02The auth decision and bundle validation are made once, above the seam, and merely enforced below it; storage-layout decisions below the seam may not require re-reading bundle semantics (guards the Forge chunking↔manifest coupling).
INV-SEAM-03A per-provider contract-test pack asserts each capability — including native relative-path resolution (provider-native on WfP, provider-absent on Forge) and the as-served CSP/sandbox posture — so a ForgeProvider that silently degrades the product's core capability fails CI.
✓
Blast radius if forced to pivot. Survives unchanged: macro components; upload pipeline; auth-gateway orchestration; lifecycle reconciliation; the entire invariant test suite; IConfluenceHost access. Rewritten/new (excluded from the reuse figure): the provider impl, the client-side reassembly engine + relative-path rewriter + CSP-on-blob, and the D1↔Forge storage mapping (unproven). The "2–4 weeks, ≥80% reuse" figure is only credible if the reassembly engine and the greenfield Connect auth gateway are scoped separately and the seam-boundary CI gate actually holds.
