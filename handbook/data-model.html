DesignDomain & Data Model

The entities, keys, and storage shapes that carry a mini-site through its life: one MiniSiteInstance per macro per page, tenant-scoped per-install secrets, immutable bundle versions for idempotent re-publish and fork lineage, an append-only hash-chained audit log, and a ProvisioningJob that tracks the async work of standing a user Worker up. State lives in D1, mirroring conf-app; the high-risk columns are app-layer encrypted.

⚑
Build gated — design documentation, not code. This page renders DESIGN.md §1.3, §2.4/§2.6, §3.3/§3.4, §5.1 and §5.6/I9. It will be enriched from BACKEND_DESIGN.md later; the schemas here are accurate to DESIGN.md as of now. Encrypted and gated items are marked inline with badges.

Entities at a glance

Five entities carry the domain. The hosting/serving identity is MiniSiteInstance, keyed by a DB-level composite (clientKey, cloudId, instanceId) — tenant binding is a database constraint, not an application-layer WHERE (INV-GW-06). Tenant secrets live in ClientInstallation, keyed by (clientKey, key) with an APP-ENCRYPTED sharedSecret. BundleVersion records the immutable storage handle that makes re-publish idempotent and fork lineage explicit; AuditLog is the append-only, hash-chained record of every security-relevant decision; and ProvisioningJob tracks the async, multi-step work of provisioning a user Worker (§3.4) — decoupled from the instance row so retries and history don't muddy it.

MiniSiteInstance — the hosting identity (§1.3)

One row per mini-site on one page. instanceId = sha256(cloudId + ":" + macroLocalId) truncated to a Worker-name-safe slug; the Connect per-macro localId + cloudId is globally unique and survives page reload, but not page copy (that is the Lifecycle model's job — §5.1/I2). workerName = ms-<instanceId>, the script name inside the dispatch namespace, and is never exposed to the client: the iframe URL only ever targets the dispatch Worker, and the instanceId → workerName mapping is server-side only.

▦
State lives in D1, mirroring conf-app: binding DB declared per-env in wrangler-*.toml, accessed through prepared-statement helpers (functions/utils/dbUtils.ts) and race-safe idempotent upserts (INSERT ... ON CONFLICT DO UPDATE). Migrations live in functions/migrations/NNNN_*.sql.

  status      TEXT NOT NULL,             -- 'staging' | 'active' | 'orphan_candidate' | 'deleted'
  scanStatus  TEXT,                      -- secret-scan / CSP verdict at publish time
  missingPasses INTEGER NOT NULL DEFAULT 0, -- reconciliation counter (§5.1/I1)
  createdAt   TEXT NOT NULL DEFAULT current_timestamp,
  updatedAt   TEXT NOT NULL DEFAULT current_timestamp,
  lastSeenAt  TEXT,                      -- updated on each authenticated view (orphan-GC input)
  -- [hardened] tenant-binding is a DB-LEVEL composite key, not an app-layer WHERE (INV-GW-06):
  PRIMARY KEY (clientKey, cloudId, instanceId)
);
CREATE INDEX IF NOT EXISTS idx_minisite_cloud  ON MiniSiteInstance(cloudId);
CREATE INDEX IF NOT EXISTS idx_minisite_orphan ON MiniSiteInstance(status, lastSeenAt);
⛨
The composite key is load-bearing. PRIMARY KEY (clientKey, cloudId, instanceId) is a DB-level constraint so instance X of tenant A cannot be queried under tenant B's valid token, and the defense cannot regress silently into an application-layer WHERE (INV-GW-06). macroInstanceId is never authoritative on its own.

Instance status lifecycle

The status column drives serve eligibility and orphan GC. A bundle is servable only when its provisioning completed atomically (bytes written and registry committed and scan passed — I10); a confirmed-absent owner advances toward deletion only after GRACE_PERIOD consecutive missing passes (I1), and an owner that cannot be confirmed present or absent is never deleted (fail-safe-retain).

ProvisioningJob — provisioning is a JOB, not a request (§3.4)

Spinning up a user Worker — WfP script-upload, Static-Assets upload of many or large files, then a reachability smoke-check — can take minutes and can exceed a single Worker request's wall-clock budget. So publish must not block on provisioning: the POST validates + stages + enqueues a job and returns 202 {jobId, statusUrl}; the actual provider.createInstance / updateBundle runs asynchronously on Cloudflare Workflows (durable execution). A ProvisionInstance workflow runs the multi-step provision — upload script, upload static assets, smoke-verify reachable via dispatch, flip MiniSiteInstance to active — with per-step durability, automatic retries + backoff, and survival across minutes and restarts.

A ProvisioningJob row tracks state, decoupled from MiniSiteInstance so retries and history don't muddy the instance row. The instance stays staging and un-servable until the workflow's final atomic step flips it to active — the gateway never serves a staging / provisioning instance (I10). The CLI and the macro-editor widget poll GET /api/jobs/{jobId} for {state, step, progress, error}; this is exactly what the Upload UI's uploading state already models — the ticking file manifest and progress numeral are the front-end of this job.

⇄
Idempotent and seam-portable. The job is keyed by Idempotency-Key + bundleHash, so a retried publish resumes the same job, never a duplicate Worker (I10). The job manager lives above the HostingProvider seam — ForgeProvider provisioning (custom-content + ~100MB/file attachment chunking + reassembly) is also long-running, so the same async manager orchestrates provider.createInstance regardless of substrate and survives the Cloudflare→Forge pivot unchanged. Rejected substrates: a plain Queue + consumer Worker (hand-rolls the step-state/retry/status Workflows gives natively — kept as fallback) and a Durable Object coordinator (overkill unless we need fan-in across steps).

ClientInstallation — tenant secrets (§2.4 / §2.6)

The Connect installed webhook delivers {clientKey, key, sharedSecret, baseUrl, ...}. The critical modeling fact: the unique key is (clientKey, key), not clientKey alone. conf-app migration 0008_fix_client_installation_unique_constraint.sql dropped UNIQUE(clientKey) and replaced it with UNIQUE(clientKey, key) — the same clientKey has multiple rows (Lite / Full app variants) with different sharedSecrets. A lookup WHERE clientKey = ? is non-deterministic; secret selection must key on (clientKey, key) where the app-variant key/aud is bound to a verified claim. A clientKey with multiple rows must never silently pick row[0] (INV-GW-04).

⛨
sharedSecret MUST be application-layer encrypted APP-ENCRYPTED. conf-app's ClientInstallation.sharedSecret is plaintext TEXT (migration 0005); Cloudflare's default at-rest encryption is not enough — a D1 read or a logged row discloses it. INV-GW-10 requires application-layer envelope / KMS-wrapped encryption of sharedSecret and the grant-signing key K_grant. Secrets and full tokens are never logged — a structurally enforced invariant (lint rule + test), because the reused conf-app code violates it.

The signed-path grant token (§2.7) — not a stored entity

Sub-resources of a multi-file bundle (app.js, style.css, assets/*) carry no Authorization header and no jwt param, and the isolated, opaque-origin sandbox (§5.3/I6) suppresses cookies. The grant token closes the sub-resource bypass without storage: on allow, the gateway mints G = base64url(payload).HMAC(payload, K_grant) with payload = {i: instanceId, c: contentId, a: accountId, cl: cloudId, exp: now + TTL}, injects a base href of /v/<instanceId>/g/<G>/ into index.html, and re-validates the HMAC + exp + permission cache on every sub-resource request. TTL = the permission-cache TTL (≤60s). The grant lives in the URL path, never in the DB.

Permission cache & KV / R2 namespaces (§2.6, I8)

Only positive allow decisions are cached, TTL ≤ 60s (configurable to 0). A cache miss is a live check, never default-allow; an outage denies even if a cached allow exists (INV-GW-11), and the circuit-breaker's default/error state is DENY. The revocation window = exp-skew + cache-TTL — worst case ~120s, not 60s — so either sign off on ~120s explicitly or set the exp skew to 0/one-sided (INV-GW-05).

⛨
Every cache and storage key MUST include cloudId (taken from the verified token, never the URL). A Cache API key that omits cloudId is a cross-tenant cache poison; the edge-cache-key-includes-cloudId property is an automated unit assertion on the cache-key builder, not pen-test-only (I8). A cloudId mismatch returns 404 (preferred over 403 to avoid an existence oracle).

BundleVersion — idempotent re-publish & fork lineage (§5.1)

A BundleRegistry / MiniSiteInstance row is keyed by (clientKey, cloudId, instanceId) with the immutable bundleId storage handle and the owning contentId (mirrors conf-app CustomContent keyed by (contentId, appId), migration 0004 [verified]). The immutable contentHash makes re-publish idempotent: an identical bundle resolves to the same bundleId, so a retried publish is a single overwrite, not a duplicate.

Copy / fork semantics (I2) eager-fork GATED

Invariant: a copied macro instance never shares a mutable bundle with its source and never overwrites it. On first authenticated render or save of a macro whose Confluence identity differs from the bundleId's recorded owner, the gateway forks — copy-on-access snapshot to a new bundleId, rebind the new contentId — and the source binding is byte-for-byte unchanged. Two pages on one mutable bundle would mean editing one silently rewrites the other; a template instantiated 50× must yield 50 independent bundles.

⇄
This is proven real in the repo: on copy "the Forge context still points at the source customContent … the save creates a fresh record with a different id" + diagram.isCopy (src/model/ContentProvider/Persistence.ts ~62-69, conf-app#170). Eager fork on a copy/export AVI event is GATED on whether avi:confluence:* copy/export events fire reliably; copy-on-access fork is the non-gated default.
⚑
Retention couplings. A bundle with any inbound un-forked copy-pointer is retained (I1c) — else GC would delete a source an un-rendered copy still depends on. DSAR erasure (I5) must transitively erase every fork descended from the target, not just the named bundleId COST backup bound GATED.

AuditLog — append-only, hash-chained (§5.6 / I9)

Invariant: every security-relevant event — auth decision, permission re-check, upload/scan, fork-on-copy, GC deletion, DSAR erasure, cross-tenant denial — is written to an append-only, tamper-evident log (timestamp, cloudId, actorAccountId, action, resourceId, outcome), retained, and queryable; no code path updates or deletes an existing row before retention expiry. The public API exposes only append + query; each record carries a prior-record hash so a tampered middle record is detectable by the verifier.

conf-app has the substrate (insert-only D1 fact tables + immutable R2 archival) [verified] but not hash-chaining. Secrets and full tokens are never in the log lint+test-enforced — conf-app's authenticate.ts:28 logs the decoded token and forge-installed.ts:34 logs the full lifecycle body (which, for classic Connect, contains the sharedSecret) [verified]; the mini-sites gateway must not inherit that.

⛨
Tail-truncation is the residual. A hash chain in mutable D1 detects middle edits but not deletion of the most recent records — an attacker with DB creds can truncate the tail undetectably. Pin the chain HEAD/length to an external append-only anchor (R2 Object Lock or a periodic external checkpoint) to make tail-truncation detectable COST. Tamper-proof (vs evident) needs WORM / Object-Lock / external SIEM COST.
⚑
Open — legal. I9 audit records contain the DSAR subject's actorAccountId; the retention basis (pseudonymize vs legitimate-interest) must be resolved rather than deferred (§5.4/I5). This is an OPEN item, not a settled column semantic.
