DeliveryGates & Positioning

No production code ships until all four blocking gates pass. This page states the gates verbatim, the honest positioning behind them (first-mover execution wedge, no durable moat, demand still n=1), the open questions that remain unresolved, and the kill rule if the gates fail.

⛔
Status: DO NOT BUILD YET. The design (this handbook) is gate-#3 work — it turns the lifecycle/security constraints into testable invariants and threat models. It is not a green light. The auth gateway is self-built ACL code of the CVE-2021-26073 / CVSS-9.1 class; a single logic error is a cross-tenant content-disclosure breach, not a degraded feature.

The four blocking gates

Every gate must pass before a single production line is written. They are ordered so a failure on an earlier gate can disqualify the whole architecture before later gates are even worth evaluating.

G1  ● GATE  DISQUALIFY RISK

Residency vs access — written admin + security sign-off from the anchor team

Written admin + security confirmation from the anchor team covering: the GitLab Pages config (access-controlled / requires a GitLab account), whether the non-technical consumers actually lack GitLab accounts, and — the load-bearing clause — residency vs access.

⚑
If they require no external processor, Cloudflare is disqualified. Files live on Cloudflare, not inside Atlassian — this is NOT "Runs on Atlassian", NOT no-egress, NOT in-boundary. A residency rejection forces the §6 Forge pivot (and only after a Forge feasibility spike first passes). Until written confirmation lands, the entire architecture is conditional.

G2  ● GATE  DEMAND

Demand-to-pay — ≥3 prospects beyond the anchor team

At least 3 prospects beyond the anchor team, each with all of:

  • a real existing mini-site,
  • a failed alternative,
  • recurring publish frequency,
  • a named approver who approved the actual external-processor architecture (residency + DPA + auth-gateway threat model),
  • a concrete commitment — paid pilot, LOI, or install approval.
ℹ
Demand from a prospect whose security would veto the external host does not count. The approval must be of the architecture as designed, not of a hypothetical in-boundary version. This gate is what turns "interesting" into "validated beyond n=1."

G3  ● GATE  THIS DOC

Pre-implementation lifecycle / security acceptance criteria signed off

The ADR + threat model + the testable invariants must be signed off before code. This handbook is the artifact that satisfies G3's production — §2 (auth gateway + STRIDE + attack tree) and §5 (the falsifiable INV-* invariants) are the document the security gate signs against. What remains is the sign-off, not the authoring.

⚑
Authored, not approved. Sign-off is a Cloud Fortified review obligation, not a self-certification — the gateway is greenfield security-critical code with zero precedent in conf-app (conf-app's request auth is Forge-only RS256/JWKS; there is no classic-Connect HS256/qsh/sharedSecret path to inherit).

G4  ● GATE  EV + KILL

EV vs shipping the next ZenUML feature + a kill criterion

An expected-value comparison against shipping the next ZenUML feature, plus a kill criterion stronger than "cheap to build." The honest re-costing (below) is the input: the project is cheap to host but expensive to make safe and compliant, so "cheap" cannot be the justification.

⚖
The kill criterion must survive the §1.5 re-cost: a realistic loaded run-rate at 10k active instances is order ~$500–700/mo (audit retention + secret-scan dominate), and the engineering cost dwarfs the infra (greenfield CVSS-9.1 gateway, recurring external pen-test, orphan GC, provable DSAR erasure). EV must clear that bar, not the $25/mo headline.

Positioning

◎
The only Confluence-native way to upload + host a multi-file bundle — a first-mover execution wedge with no durable moat. Forge Custom UI and the 4.5k–8.7k-install incumbents could copy it; governance is table stakes, not a differentiator. Demand is n=1 (the anchor team), unvalidated beyond — hence the gates.

Three honest framings sit under that one-liner, and none of them is softened:

FramingWhat it meansConsequence
Execution wedge, not a moatThe capability is reproducible by incumbents with Forge Custom UI and a few weeks of work.Value is in being first and executing the governance well — there is no defensible lock-in to rely on.
Governance is table stakesOrphan GC, tenant isolation, DSAR erasure, audit, CSP/sandbox are required just to ship — they don't win the market.The expensive invariants are the cost of entry, not the prize. They belong behind the paid tier (open question #12).
Demand is n=1One anchor team wants it; nothing beyond it is validated.G2 exists precisely to retire this risk before building. n=1 is the reason there are gates at all.

Tiering / billing decision (#12) in detail

Two-tier pricing (Standard + Advanced) is supported on the Marketplace via app editions — but only for Paid-via-Atlassian (PvA) apps. The mechanism depends on the hosting path:

PathTiering mechanismCleanliness
ForgeSet licensing.editionsEnabled: true; gate features off getAppContext().license.capabilitySet ("capabilityadvanced" vs Standard/null).CLEAN — runtime gives it to you.
CloudflareThe dispatch Worker runs outside the Forge runtime → no capabilitySet. Either (a) the auth gateway fetches + caches the Marketplace/Connect license per cloudId as one more server-side check, or (b) bill outside Atlassian (own metering keyed by cloudId).EXTRA WORK — one more gateway check or a billing system.
◎
The natural tier split maps onto the §1.5 cost-drivers: put audit-log access (I9), larger bundle/instance limits, configurable secret-scan policy (I7), CLI/MCP publish (Flow A), and SLA support in Advanced — so the expensive invariants sit behind the paid tier.

The honest cost reality

The premise was "~$25/mo, solo-buildable." That prices only the hosting. The acceptance criteria add a per-view + per-publish variable layer the premise ignored — every line below was $0 in the original framing.

💰
Cheap to host, expensive to make safe / compliant. The premise does not survive intact. A realistic loaded run-rate at 10k active instances under real traffic is order ~$500–700/mo (audit retention + secret-scan dominate), and the engineering cost dwarfs the infra: the greenfield auth gateway (CVSS-9.1 class), a recurring Cloud Fortified review + external pen-test (a four-figure recurring item, not a rounding error), orphan GC, and provable DSAR erasure are each multi-week security-critical workstreams. This belongs in gate #4.

The kill rule

⛔
If the gates do not pass → ship the next ZenUML feature instead. That is the default, not a fallback. The gates exist to make "don't build this" the easy, pre-committed outcome when the evidence isn't there — n=1 demand, an un-signed threat model, a residency veto, or an EV that loses to the next ZenUML feature each independently route to this rule.
