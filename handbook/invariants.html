DeliveryLifecycle & Invariants

Ten acceptance criteria, restated as falsifiable invariants. Each names the threat it guards, the concrete test (unit / integration / E2E layer + assertion), and the honest residual risk. This is the gate-#3 deliverable — the security & correctness contract the build must satisfy before it is unlocked.

✓
This page is the gate-#3 acceptance artifact. Per the design, "Testable invariants & threat models" is itself the gate-#3 deliverable. Every constraint below maps to a falsifiable invariant with a named test layer and a stated residual risk. The auth-gateway invariants (INV-GW-*) referenced in I3/I4/I8 are the detailed expansion documented under Auth Gateway & Threat Model.
⚑
Status: BUILD GATED. This is design documentation, not shipped code. Items tagged GATED depend on unverified platform behaviour (chiefly avi:confluence:* event reliability and Cloudflare backup/erasure semantics). Items tagged COST carry recurring cost or ops that the "~$25/mo, solo-buildable" premise does not absorb — a finding, not a blocker. [hardened] markers denote invariants added or strengthened after the red-team / completeness review.

Invariant register (I1–I10)

The full set at a glance. Statement, the threat it guards, the layered test, and residual risk. Tags surface GATED and COST items inline; severity / cross-tenant items carry a RISK tag.

I1 Orphan-GC reconciliation: a bundle is retained iff a reachability check confirms its owning contentId exists and still references the bundleId; absent ≥ GRACE passes ⇒ deleted; unconfirmable (API error) ⇒ never deleted, never served past the gate. (a) storage leak / cost — no delete webhook; (b) worse: wrong deletion of a live bundle when a transient outage looks like "deleted". unit state-machine table; integration over local D1 + in-memory provider; E2E lifecycle/orphan-gc.spec.ts create→delete page→reconcile→assert 404 + row gone. Eventually consistent (survives up to GRACE × interval); unbounded leak during a sustained Confluence outage (correct by design).
I2 Copy / duplicate / template / space-export: a copied instance never shares a mutable bundle with its source and never overwrites it. Foreign-identity access ⇒ fork (copy-on-access snapshot → new bundleId, rebind); source byte-for-byte unchanged. Silent shared mutable state — proven real in the repo: a copy still points at the source customContent; editing one page silently rewrites the other. unit forkOnCopy.spec.ts (fork-on-foreign-access + idempotent); integration 10 contentIds → 10 bundleIds, source identical; E2E via macroDuplication.ts — mutate copy, assert source unchanged. A never-rendered copy keeps a dangling pointer (→ I1c). Eager fork on copy/export event is GATED on whether avi:confluence:* events fire reliably. GATED
I3 Permission-cache staleness / fail-closed serve: a protected request is served only if a decision for (viewerAccountId, contentId, "read") is affirmative under a cache entry younger than PERM_CACHE_TTL; absent / expired / erroring ⇒ 403, never the bundle; revocation clears within ≤ TTL. Serving revoked content — direct disclosure, since permissions are app-enforced. unit (allow+fresh→serve; allow+expired→re-check; recheck throws→403 — the key assertion; deny→403; revoke→403); integration flip allow→deny; E2E revoke-no-serve + pen-test rider (direct user-Worker probe must fail, INV-GW-14). ≤ TTL window for a just-revoked user; shrinking to 0 = a synchronous Confluence call per asset (COST); event-driven invalidation GATED on AVI reliability. I4 Token expiry mid-view: every protected request carries a currently-valid grant derived from a currently-valid Atlassian JWT (signed-path grant, §2.7); expired / malformed ⇒ 401 with a re-auth signal, never cached-as-allow, never served. A long-open embed whose initial token expires, naively kept serving (bypass) or crashing. unit (valid→pass; exp past→401; bad sig→401; qsh mismatch→401; wrong aud/app→401, fixed clock); integration exp=now+2s → 200 at T+0, 401 at T+3s, no bytes; E2E hold-open → asset-fetch → re-auth, incl. parent-eviction path [hardened]. Clock skew needs clockTolerance leeway composing with cache TTL (~120s) — sign off or set to 0. Eviction fails closed (acceptable).
I5 GDPR / DSAR erasure on the external host: a DSAR results within SLA in provable deletion of all copies of the bundle bytes + metadata across every store, post-deletion fetch returns 404, and a deletion record exists. [hardened] erasure transitively erases every I2 fork descended from the target. Inability to honor erasure because data lives externally with caches / backups we don't account for — a GDPR / DPA blocker. unit enumerate listStoresHolding(bundleId) + assert a delete per store + a deletion certificate; integration write→populate edge cache→erase→404 + purge + row gone; E2E full-tenant erasure → every prior URL 404. Backups: "bytes exist nowhere" may only hold after backup-retention expiry (bounded delay must be in the DPA); edge purge is async; audit logs carry the DSAR subject's actorAccountId — retention basis is OPEN (legal). COST GATED
I6 Malicious uploaded JS — sandboxing / CSP / origin isolation: per-bundle origin isolation [hardened — HARD, not "ideally"], opaque-origin iframe sandbox (no allow-same-origin vs control plane), pinned restrictive CSP baseline, no trusted-origin authority leak. All four must hold. Attacker JS from a trusted-looking domain → XSS, credential phishing, malware, clickjacking. We host active bundles, so we contain — we cannot sanitize JS away. unit every response has CSP + nosniff + correct Content-Type + bundle-origin ≠ control-plane; integration hostile fixture (top.location, document.cookie, exfil fetch blocked) + four same-tenant fixtures [hardened]; E2E in a real embed. PROVIDER-SPECIFIC: containment ≠ content-safety (a cryptominer / phish within its own sandbox needs I7 + abuse policy). CSP baseline is GATED but pinned to stay falsifiable. RISK GATED
I7 Secret-leak detection: every upload is scanned before it is servable; a detected secret blocks go-live (or quarantines), notifies the uploader, and is audit-logged. "Made servable" never precedes "scan completed". [hardened] a scanner-version bump re-quarantines already-live bundles until re-scan (I7↔I10 coupling). An AI-generated bundle with a baked-in key / token / PEM becomes readable by everyone with page access and persists externally — a customer incident + processor liability. unit fixture table (AWS / Google / JWT / PEM / password= detected; UUID / hex-color / base64-image not flagged); integration secret bundle → quarantined, not servable; E2E blocked-with-reason. Scanning is best-effort (novel / obfuscated / encrypted secrets evade; false positives frustrate) — stated to customers as best-effort, not a guarantee. Per-upload CPU. COST
I8 Tenant isolation (cross-install / cross-space / cross-instance): no request authenticated for cloudId=A can read / list / enumerate / provision any bundleId owned by cloudId=B; every storage key, registry row, permission check, dispatch route and edge-cache key is namespaced by cloudId from the verified token; a mismatch ⇒ 404 (avoids an existence oracle). Cross-tenant exposure — the highest-severity failure and a guaranteed Cloud-Fortified blocker. unit A-token→B-resource→404 + key-builder cannot produce a B key from an A context; integration exhaustive A→B (direct / guessed / enumeration, all three scopes distinct); E2E two-tenant deny + audit entry. [hardened] edge-cache-key-includes-cloudId promoted to an automated unit assertion (a cache key omitting cloudId is cache poison). PROVIDER-SPECIFIC: a WfP namespace-isolation platform bug is vendor risk — disclose in the review. RISK
I9 Audit logs: every security-relevant event (auth decision, permission re-check, upload / scan, fork-on-copy, GC deletion, DSAR erasure, cross-tenant denial) is written to an append-only, tamper-evident log (timestamp, cloudId, actorAccountId, action, resourceId, outcome); no path updates / deletes a row before retention expiry. Inability to investigate / prove an incident; an attacker (or bug) erasing tracks. unit API exposes only append+query + each record carries a prior-record hash; integration tamper a middle record → verifier detects; E2E a denied cross-tenant access + a GC deletion both appear. [hardened] pin chain HEAD/length to an external append-only anchor for tail-truncation detection. Tamper-proof (vs evident) needs WORM / Object-Lock / external SIEM; log volume at 10k instances is non-trivial. COST
I10 Partial-failure recovery (no half-state serves): a bundle is servable iff provisioning completed atomically (bytes fully written and registry committed and scan passed); any mid-way failure leaves it not servable, idempotently resumable or cleanly rolled back — never a half-deployed Worker serving a truncated bundle. WfP Worker-create + asset-upload + registry-write + scan has no native cross-system transaction; a gap could serve partial assets, a row pointing at nothing, or an unscanned "live" bundle (bypassing I7). unit inject failure after each step → isServable=false + re-run idempotent; integration provider throws mid-write → "not ready", cleanup/retry reconciles (ties to I1); E2E kill-mid-deploy → "deploying/failed" not a broken render → retry succeeds. [hardened] add a serve-time content-hash check for committed-but-corrupt bytes; retry-storm backoff / circuit-breaking as an invariant; the I7 re-scan-on-version-bump tie-in. COST

Sub-model A — Lifecycle (I1 & I2)

The lifecycle pair shares one structure: a BundleRegistry / MiniSiteInstance row keyed by (clientKey, cloudId, instanceId), carrying the immutable bundleId storage handle and the owning contentId. This mirrors conf-app's CustomContent keyed by (contentId, appId) (migration 0004, verified). Confluence emits no reliable per-macro delete event, so retention is driven by a standing reconciliation cron (the repo's aggregate-events.ts x-cron-secret → purgeOldEvents is the shape; the external-reachability probe is net-new).

I1 — Orphan-GC reconciliation correctness Lifecycle

Invariant. A stored bundle is retained iff a reachability check confirms its owning contentId still exists and still references that bundleId; a bundle confirmed-absent for ≥ GRACE_PERIOD consecutive passes is deleted; a bundle whose owner cannot be confirmed present or absent (API error / outage) is never deleted (fail-safe-retain) and never served beyond the permission gate (fail-closed-serve, I3).

Threat. (a) storage leak / cost from no delete webhook; (b) worse — wrong deletion of a live bundle because a transient outage looked like "deleted."

I1a–I1d — red-team-hardened additions

Add-onWhat it pins down
I1a — owner-identity stability Reachability must survive page move / rename / space-relocation, and a restore-from-trash that returns a NEW contentId must re-bind, not orphan (a single exists probe is insufficient). Default: treat trash as present (retain) until purge.
I1b — registry-store outage If BundleRegistry / D1 is itself unreadable, serve = 403 (cannot confirm ownership ⇒ deny) — distinct from the permission-outage case (I3).
I1c — inbound copy-pointer retention A bundle with any inbound un-forked copy-pointer (I2) is retained — else GC deletes a source an un-rendered copy still depends on, making the copy permanently un-forkable.
I1d — reconcile blast-radius cap The reconcile endpoint is triggered by a shared secret (x-cron-secret pattern); a leaked secret = attacker-driven mass deletion. The reaper cannot delete > N bundles/pass without a second control.

Test. unit table over the state machine (present→retained; absent×(GRACE-1)→retained; absent×GRACE→deleted once; apiError→counter unchanged, retained); integration over local D1 + in-memory provider; E2E tests/e2e-tests/tests/lifecycle/orphan-gc.spec.ts create → delete page → reconcile → assert 404 + row gone.

Residual. Eventually consistent (a bundle survives up to GRACE × interval); an unbounded leak during a sustained Confluence outage (correct by design).

I2 — Copy / duplicate / template / space-export semantics Lifecycle GATED

Invariant. A copied macro instance never shares a mutable bundle with its source and never overwrites it. On first authenticated render or save of a macro whose Confluence identity differs from the bundleId's recorded owner, the gateway forks (copy-on-access snapshot → new bundleId, rebind new contentId); the source binding is byte-for-byte unchanged.

Threat. Silent shared mutable state — proven real in the repo: on copy "the Forge context still points at the source customContent … the save creates a fresh record with a different id" + diagram.isCopy (src/model/ContentProvider/Persistence.ts ~62-69, conf-app#170). For a hosted bundle, two pages on one mutable bundle = editing one silently rewrites the other; a template instantiated 50× must yield 50 independent bundles.

Test. unit forkOnCopy.spec.ts (fork-on-foreign-access + idempotent — no re-fork); integration (10 distinct contentIds → 10 distinct bundleIds, source byte-identical); E2E reusing tests/e2e-tests/helpers/macroDuplication.ts (duplicateMacroSamePage, copyMacroToNewPage) — mutate the copy, assert source unchanged (UI evidence).

Residual. A never-rendered / never-saved copy keeps a dangling pointer (covered by I1c). Bulk space-export captures the macro body but not the external bundle → the importing site renders a "bundle not available in this site" placeholder (cross-tenant, I8). Eager fork on a copy/export AVI event is GATED on whether avi:confluence:* copy/export events fire reliably.

Sub-model B — Access / Auth gateway (I3, I4, I8)

The primary pen-test targets. These are the detailed expansion point for the INV-GW-* family in the Auth Gateway page — the IDOR class here is CVE-2021-26073 territory, and a cross-tenant break (I8) is the highest-severity failure, a guaranteed Cloud-Fortified blocker.

I3 — Permission-cache staleness / fail-closed serving

Invariant. A protected request is served only if a decision for (viewerAccountId, contentId, "read") is currently affirmative under a cache entry younger than PERM_CACHE_TTL; absent / expired / erroring ⇒ 403, never the bundle; revocation invalidates within ≤ TTL. conf-app's src/model/page/CheckPermission.ts posts to /permission/check and returns false on any error (verified) — that fail-closed default is the load-bearing pattern. The single most important assertion: a re-check that throws resolves to 403.

Residual. A ≤ TTL window for a just-revoked user (shrinking to 0 means a synchronous Confluence call per asset → latency / rate-limit / cost COST); event-driven invalidation GATED on AVI reliability. Confluence's effective permission (page restrictions vs space perms vs inherited) is subtle — the pen-test must probe restriction-inheritance edge cases.

I4 — Token expiry mid-view

Invariant. Every protected request carries a currently-valid grant derived from a currently-valid Atlassian JWT (§2.7); expired / malformed ⇒ 401 with a re-auth signal, never cached-as-allow, never served; a view outliving token lifetime degrades to a re-auth prompt, not open access. The signed-path grant + refresh-and-retry mechanism (§2.7) closes the previously-unimplementable gap.

Residual. Clock skew needs clockTolerance leeway, but it composes with cache TTL (~120s, §2.4) — sign off or set to 0. Re-auth inside a sandboxed iframe relies on messaging the parent; if the parent is evicted the view fails closed (acceptable) — add an E2E for the eviction path, not just the happy path [hardened].

I8 — Tenant isolation (cross-install / cross-space / cross-instance) RISK

Invariant. No request authenticated for cloudId=A can read / list / enumerate / provision any bundleId owned by cloudId=B; every storage key, registry row, permission check, dispatch route, and edge-cache key is namespaced by cloudId (taken from the verified token, never the URL); a cloudId mismatch ⇒ 404 (preferred over 403 to avoid an existence oracle). conf-app already namespaces by cloudId everywhere (license:${cloudId}:${spaceKey} KV, page-snapshots/${cloudId}/... R2, verified).

[hardened]. Move the edge-cache-key-includes-cloudId property from manual-pen-test-only to an automated unit assertion on the cache-key builder (identical to the storage-key builder test) — a Cache API key omitting cloudId is a cross-tenant cache poison. Integration must assert all three scopes (cross-install / cross-space / cross-instance) distinctly, by direct id, guessed id, and enumeration.

Residual PROVIDER-SPECIFIC. A WfP namespace-isolation platform bug is vendor risk — disclose in the review.

Sub-model C — Uploaded-content security (I6, I7)

We host active bundles (that is the product), so we cannot sanitize JS away — we contain it. This is the area with the least precedent in conf-app (which renders its own diagram output, not arbitrary uploaded JS), hence the highest-novelty risk and a named Cloud-Fortified focus.

I6 — Malicious uploaded JS: sandboxing / CSP / origin isolation RISK GATED

Invariant (all four must hold):

  1. Per-bundle origin isolation [hardened — HARD, not "ideally"]: every bundle is served from an origin distinct from the control-plane origin and distinct per bundle (not merely per tenant). Same-tenant bundle A must not be able to script bundle B, fetch B's assets, read B via a co-framed DOM, or poison a shared-origin edge-cache key. Uploaded JS never executes on the origin holding the gateway's auth state.
  2. Iframe sandbox: <iframe sandbox="allow-scripts allow-forms allow-popups-to-escape-sandbox"> — no allow-same-origin relative to the control plane (opaque origin; this is why §2.7 uses path grants, not cookies).
  3. CSP [GATED baseline — but pinned now]: the bundle response carries a restrictive CSP, pinned now so I6 is falsifiable: frame-ancestors = the Confluence / app embed origin only (anti-clickjacking); script-src self + bundle origin (no inline unless hashed); connect-src / form-action restricted to prevent exfiltration / phishing via the trusted origin; never frame-ancestors *. Security review may tighten, not loosen.
  4. No trusted-origin authority leak: a bundle response never sets or exposes the gateway's session state / K_grant to bundle-origin script.

Test. unit (every response has the CSP, nosniff, correct Content-Type, bundle-origin ≠ control-plane); integration with a hostile fixture attempting top.location, document.cookie, exfil fetch → all blocked, plus the four same-tenant fixtures [hardened]: A-scripts-B, A-fetches-B-assets, shared-origin-cache-poison, B-readable-via-co-framed-DOM; E2E in a real embed (console / UI evidence).

Residual PROVIDER-SPECIFIC. Containment ≠ content-safety (a cryptominer / phish within its own sandbox needs I7 + abuse policy, not CSP). Forge-equivalent: platform CSP / sandbox shifts much of this to Atlassian — a genuine posture advantage if the pivot is forced (but reassembly re-expands the in-iframe surface, §6).

I7 — Secret-leak detection COST

Invariant. Every upload is scanned before it is servable; a detected secret blocks go-live (or quarantines), notifies the uploader, and is audit-logged (I9); "made servable" never precedes "scan completed". [hardened] I7↔I10 coupling: a scanner-version bump must re-quarantine already-live bundles until re-scan.

Threat. An AI-generated bundle with a baked-in key / token / PEM becomes readable by everyone with page access and persists externally — a customer incident + a processor liability.

Test. unit fixture table (AWS / Google / JWT / PEM / password= detected; UUID / hex-color / base64-image not flagged); integration (secret bundle → quarantined, not servable); E2E blocked-with-reason.

Residual. Scanning is best-effort (novel / obfuscated / encrypted secrets evade; false positives frustrate) — state to customers as best-effort, not a guarantee; we notify, we do not rotate / revoke the leaked secret. Per-upload CPU COST.

Sub-model D — Operations & compliance (I5, I9, I10)

I5 — GDPR / DSAR erasure on the external host COST GATED

Invariant. A DSAR / erasure request results, within SLA, in provable deletion of all copies of the affected bundle bytes + metadata across every store (primary object store, derived caches, backups) such that a post-deletion fetch of any prior bundleId returns 404 and an auditable deletion record exists. No soft delete satisfies it for the bytes. [hardened] fork-lineage transitivity: erasure must transitively erase every I2 fork descended from the target, not just the named bundleId (listStoresHolding is per-bundleId and would miss forks).

Residual (significant). Backups — true "bytes exist nowhere" may be achievable only after backup-retention expiry (GDPR permits documented, bounded backup-rotation delays — must be verified with Cloudflare and stated in the DPA, COST GATED); edge purge is async ("purge issued" ≠ "globally complete" — set a served-bytes-after-erasure bound); I9 audit logs contain the DSAR subject's actorAccountId — resolve the retention basis (pseudonymize vs legitimate-interest) rather than deferring (OPEN — legal).

I9 — Audit logs COST

Invariant. Every security-relevant event (auth decision, permission re-check, upload / scan, fork-on-copy, GC deletion, DSAR erasure, cross-tenant denial) is written to an append-only, tamper-evident log (timestamp, cloudId, actorAccountId, action, resourceId, outcome), retained, and queryable; no code path updates / deletes an existing row before retention expiry. conf-app has the substrate (insert-only D1 fact tables + immutable R2 archival, verified) but not hash-chaining.

[hardened] tail-truncation. A hash chain in mutable D1 detects middle edits but not deletion of the most recent records — an attacker with DB creds can truncate the tail undetectably. Pin the chain HEAD / length to an external append-only anchor (R2 Object Lock or a periodic external checkpoint) to make tail-truncation detectable COST.

Residual. Tamper-proof (vs evident) needs WORM / Object-Lock / external SIEM COST; log volume at 10k instances is non-trivial.

I10 — Partial-failure recovery (no half-state serves) COST

Invariant. A bundle is servable iff its provisioning completed atomically (bytes fully written and registry committed and scan passed); any mid-way failure leaves it not servable, and the transaction is idempotently resumable or cleanly rolled back — never a half-deployed Worker serving a truncated bundle. conf-app documents exactly this idempotent-resume discipline (forge-custom-content.ts:44-67, "recover from prior requests that inserted the version but failed before completing the write", verified).

[hardened] additions. (a) a serve-time content-hash integrity check for the committed-but-corrupt case (write succeeded, bytes wrong — not caught by transaction logic) COST; (b) retry-storm backoff / circuit-breaking as an invariant (avoid amplifying a Cloudflare outage); (c) the I7 re-scan-on-version-bump tie-in.

Cross-cutting invariant — abuse / DoS bounds RISK

⚠
This invariant was missing entirely (§5.5, [hardened]). The system bounds upload size / count, fork amplification (a template instantiated N× = N copyBundle calls — cap per pass / tenant), reconcile-endpoint abuse (I1d blast-radius cap), and bundle-serve bandwidth per tenant. Compromise of any control secret (x-cron-secret, K_grant, PAT) must not enable mass deletion or forged grants — secrets are rotated, envelope-encrypted, never logged, and the reconcile path requires a second control beyond the shared secret.
