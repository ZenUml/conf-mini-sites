DesignSequence Flows

The five core runtime flows, drawn end to end: how a viewer's request is authorized and served, how a bundle is published, how installs come and go, how orphaned bundles are reaped, and how a DSAR erasure is proven. Each flow is fail-closed by construction.

⚑
Status: build is GATED. These diagrams are design documentation, not shipped behavior. The SERVE path crosses the auth gateway — a single logic error there is a cross-tenant content-disclosure breach (CVSS-9.1 class, CVE-2021-26073 territory), not a degraded feature. Treat every step as a checked invariant, not an aspiration.

One vocabulary across all diagrams: Viewer (the browser iframe), DispatchGateway (the auth gateway / dispatch Worker — the single network entry point), UserWorker (the non-routable per-instance Worker that returns raw bytes), D1 (instance registry + audit), KV (encrypted install secrets + permission cache), R2 (bundle object store + immutable audit archive), ConfluenceAPI (permission/check + reachability), CronReaper (the scheduled orphan reaper), and Provider (the HostingProvider seam, today CloudflareWfPProvider).

1 · SERVE — viewer to bytes, every byte authorized

The entrypoint iframe loads GET /v/<instanceId>?jwt=<connect-jwt> — the single network entry point. The gateway authenticates, binds the resource to a server-verified context, calls permission/check(read), and only then mints a short-lived signed-path grant and serves index.html with an injected <base href="/v/<instanceId>/g/<grant>/"> plus the CSP/sandbox headers. Every relative sub-resource then re-enters the gateway, which re-validates the grant and re-checks the permission cache before dispatching to the user Worker. On the Cloudflare path the user Worker is never reached on a deny, so revoked content cannot leak even for a half-second.

⛨
This per-request check is the only thing standing between an authenticated user and any bundle's bytes. Unlike conf-app — where Confluence itself authorizes content fetched with the user's own OAuth token — here one skipped check = disclosure. It must be provably non-bypassable on 100% of asset paths, sub-resources included (INV-GW-07).

The grant token solves the sub-resource bypass: sub-resource loads carry no Authorization header and no ?jwt=, and the required sandbox without allow-same-origin gives the iframe an opaque origin that suppresses cookies. The HMAC-signed grant in the path is sent on same-document loads regardless of origin opacity, is unforgeable, and is replay-bounded by a short TTL and a per-request permission re-check.

⏱
Revocation window composes. Worst-case window = exp-skew + positive-cache-TTL. With ≤60s + ≤60s that is ~120s, not 60s — either sign off ~120s explicitly or set the exp skew to 0 so the signed-off figure is true (INV-GW-05). The circuit-breaker's default/error state is DENY: a breaker that trips open would be a disclosure (INV-GW-11).

2 · PUBLISH — one pipeline, no second door, async provisioning

Both entry points — Flow A (CLI / MCP mini_sites.publish, auth: OAuth 3LO token mapped to a real account) and Flow B (drag-and-drop in the macro editor, auth: the in-iframe Connect JWT of the uploading viewer) — funnel into one shared upload pipeline (pipeline/uploadPipeline.ts). The ordering is load-bearing: authn → authz → validate → secret-scan → posture → atomic commit. A bundle that fails any step never reaches status=live. The pipeline ends at the HostingProvider seam, so a Cloudflare→Forge pivot is a provider swap, not a publish-path rewrite.

Publish no longer blocks on provisioning (§3.4). Spinning up a user Worker — WfP script upload, Static-Assets upload of many or large files, then a reachability smoke-check — can take minutes and would exceed a single Worker request's wall-clock budget. So the publish request only validates + stages, then enqueues a provisioning job and returns 202 {jobId, statusUrl}. The actual provisioning runs asynchronously in a durable ProvisionInstance workflow (Cloudflare Workflows), tracked by a decoupled ProvisioningJob row (state queued → provisioning → active | failed), and the CLI / macro-editor widget polls GET /api/jobs/{jobId} until the state is active or failed. The job manager lives above the HostingProvider seam, so the same async model survives the Forge pivot (Forge provisioning is also long-running).

[GATED] OAuth client registration. Whether a public OAuth client with the device grant and the exact scopes (write:confluence-content, read:confluence-content.permission, custom-content write) can be registered is a vendor capability to confirm before building Flow A. ⏱
Why async — and how it stays safe. The chosen substrate is Cloudflare Workflows (durable execution: per-step state, automatic retries + backoff, survival across minutes and restarts). The fallback if Workflows is unavailable in an env is a Queue + consumer Worker that hand-rolls the same step-state/retry/status. The instance row stays staging and un-servable until the workflow's final atomic step flips it to active — the gateway never serves staging/provisioning (I10). The job is idempotent (Idempotency-Key + bundleHash → a retried publish resumes the same job, never a duplicate Worker), a failed job leaves no servable bytes, and the orphan reaper reaps instances/jobs stuck in provisioning/staging past a TTL (ties I1 ↔ I10). Per-step backoff/circuit-break avoids amplifying a WfP outage.

3 · INSTALL / UNINSTALL — the secret's origin and end

The Connect installed webhook delivers {clientKey, key, sharedSecret, baseUrl, …}. The handler must verify the install callback itself — an asymmetric JWT signed by Atlassian's published Connect install keys (iss = connect-install-keys.atlassian.com, RS256 against Atlassian's CDN JWKS) — and pin aud to this app's own baseUrl/identity, else a valid Atlassian-signed install JWT minted for a different app could seed or overwrite an install record. First install is gated solely on the verified asymmetric signature — never "no record yet ⇒ trust." The secret is stored application-layer encrypted (Cloudflare's default at-rest encryption is not enough; conf-app stored it as plaintext TEXT). On uninstalled, the secret is deleted and the tenant's resources torn down.

⛨
An unverified installed that writes a sharedSecret is an account-takeover primitive (INV-GW-10). conf-app's uninstalled.ts tolerates a missing auth header as a "Connect migration bridge" — the mini-sites gateway must NOT copy that tolerance to the installed path. Reinstall overwrites the secret atomically; tokens under the old secret fail closed during rotation.

4 · RECONCILE / GC — the orphan reaper, fail-safe-retain

Connect emits no reliable per-macro delete event (Design Constraint #1), so deletion is two-tier: the uninstalled hook tears down a whole tenant (flow 3), and a scheduled cron Worker — the orphan reaper — reconciles per-instance. The reaper enumerates instances, probes Confluence reachability for each owning contentId, and drives a state machine on missingPasses: present → retained; absent for < GRACE → retained, counter++; absent for ≥ GRACE → deleted once; API error → counter unchanged, retained (fail-safe-retain). A bundle whose owner cannot be confirmed present or absent is never deleted.

⚑
Blast-radius cap (I1d). The reconcile endpoint is triggered by a shared secret (x-cron-secret pattern); a leaked secret = attacker-driven mass deletion. The reaper cannot delete > N bundles per pass without a second control. The probe also survives page move/rename/space-relocation, and a restore-from-trash returning a NEW contentId must re-bind, not orphan (I1a). A bundle with any inbound un-forked copy-pointer is retained (I1c).

5 · DSAR ERASURE — provable deletion across every store

A DSAR/erasure request must, within SLA, result in provable deletion of all copies of the affected bundle bytes and metadata across every store — primary object store, derived edge caches, backups — such that a post-deletion fetch of any prior bundleId returns 404 and an auditable deletion certificate exists. No soft delete satisfies it for the bytes. Critically, erasure must transitively erase every fork descended from the target (I2 fork lineage), because listStoresHolding is per-bundleId and would otherwise miss forks.

▦
Every one of these flows writes to the append-only, tamper-evident audit log (I9): auth decisions, permission re-checks, uploads/scans, fork-on-copy, GC deletions, DSAR erasures, and cross-tenant denials. Secrets and full tokens are never logged — a structurally enforced invariant (lint rule + test), because the reused conf-app code violates it. The full constraint set lives in Lifecycle & Invariants.
